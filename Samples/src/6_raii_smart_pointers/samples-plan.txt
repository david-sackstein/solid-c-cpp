6_raii_smart_pointers — sample plan
====================================

Shared helper (all samples)
---------------------------
File: TracedResource.h (~10 lines)
A tiny class with ctor/dtor that prints "[Resource #N acquired]" / "[Resource #N released]".
Every runnable sample uses it so live demos show acquisition and release on stdout.

CMake: one executable per sample folder (raii_01 … raii_12), each with its own main.cpp.


Slide 1–2 — Title / Agenda
--------------------------
No code sample.


Slide 3 — The resource management problem
-----------------------------------------
Folder: 1_manual_leak/
File: main.cpp (~12 lines)

What it shows:
- Raw new without matching delete when an early return skips cleanup.
- A second path in the same program uses the same helper with manual delete on all paths.

Expected output (run: ./raii_01):
  [Resource #1 acquired]
  leaving early — no release          <-- leak; no matching "[Resource #1 released]"
  [Resource #2 acquired]
  [Resource #2 released]

Slide copy note: first block is the problem; second block is the manual fix the slides say is error-prone.

Slide 6 — Ownership contracts
-----------------------------
Folder: 4_ownership_contracts/
File: main.cpp (~18 lines)

What it shows:
- Unique: move transfers ownership; source prints null after move.
- Shared: two owners, use_count() == 2; last owner gone triggers release.

Expected output:
  unique: before move, owner holds resource
  unique: after move, source is null
  [Resource #1 acquired]
  shared: use_count = 2
  shared: use_count = 1
  [Resource #1 released]

Slide 11 — weak_ptr
-------------------
Folder: 8_weak_ptr/
Files: Node.h (~20 lines), main.cpp (~20 lines)

What it shows:
- Cycle with shared_ptr only: nodes destroyed but resource never released (use_count stays > 0 at program end — print alive count).
- Same graph with weak_ptr back-reference: all resources released cleanly.
- .lock() returns empty shared_ptr after owner is gone.

Expected output (weak_ptr version — default run):
  [Resource #1 acquired]
  child sees parent: parent
  parent scope ends
  [Resource #1 released]
  lock after release: expired

Separate target or //ifdef CYCLE_DEMO for the leaking shared_ptr version (comment in output: "resource still alive — leak").


Slide 12 — Best practices (make_* , never double-wrap)
------------------------------------------------------
Folder: 9_make_unique/
File: main.cpp (~12 lines)

What it shows:
- Prefer std::make_unique / std::make_shared (single allocation path, prints once).
- Commented anti-pattern: two shared_ptr from same raw pointer → double-delete (kept in slides as dead code, not compiled).

Expected output:
  [Resource #1 acquired]   // make_unique
  [Resource #1 released]
  [Resource #2 acquired]   // make_shared
  [Resource #2 released]


Slide 13 — Best practices (default unique, shared when needed)
--------------------------------------------------------------
Folder: 10_when_to_share/
File: main.cpp (~16 lines)

What it shows:
- Factory returns unique_ptr by default (zero ref-count overhead).
- Two containers each holding shared_ptr to the same object — use_count == 2, justified shared ownership.

Expected output:
  factory returns unique_ptr
  [Resource #1 acquired]
  [Resource #1 released]
  [Resource #2 acquired]
  registry A and B share, use_count = 2
  [Resource #2 released]


Slide 14 — Best practices (passing and lifetimes)
-------------------------------------------------
Folder: 11_passing/
File: main.cpp (~18 lines)

What it shows:
- use(const TracedResource&) — non-owning, no overhead.
- find(TracedResource*) — optional/nullable observer.
- share(std::shared_ptr<TracedResource>) — only when callee takes shared ownership (use_count increments).

Expected output:
  [Resource #1 acquired]
  observe via const&: ok
  observe via *: ok
  observe via nullptr: skipped
  share: use_count now 2
  [Resource #1 released]
